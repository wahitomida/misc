
# 第1章 プロジェクト概要

---

## 1.1 AI Orchestra とは

AI Orchestra は、**複数の AI エージェントが役割を持って議論し、研究者のアイデアや実装を多角的にブラッシュアップする CLI ツール**です。

オーケストラに例えると：

```
🎼 指揮者（Orchestrator）
├── 議論の計画を立てる
├── 適切な奏者（AI）を選ぶ
├── テンポと方向性を制御する
└── 最終的に結果をまとめる

🧮🔬🤖📚😈🎯 奏者（AI エージェント）
├── それぞれ得意分野と性格を持つ
├── 指揮者の指示を受けて発言する
├── 他の奏者の発言に反応して議論を深める
└── 自分の貢献を自己評価する
```

1人の研究者が考えると、自分の専門バイアスや思考の癖に縛られがちです。AI Orchestra は**理論屋・実験屋・実装屋・文献屋・穴探し・鳥の目**という異なる視点を持つ AI を同時に動かすことで、研究室のホワイトボード前での活発な議論を再現します。

## 1.2 解決する課題

研究者が日常的に直面する以下の課題を解決します。

### アイデア検討フェーズの課題

| 課題 | 現状 | AI Orchestra による解決 |
|---|---|---|
| 壁打ち相手がいない | 1人で考えて堂々巡り。指導教員や同僚は忙しい | いつでも5分で多角的議論が回せる |
| 専門バイアス | 自分の得意手法に偏る | 理論・実装・実験・文献の全方位から検証 |
| 穴に気づけない | 前提を疑わず進めてしまう | 😈穴探しが前提を崩し、反例を構築する |
| 先行研究の見落とし | サーベイが不十分なまま進行 | 📚文献屋が関連研究を網羅的に提示 |
| 実験計画の甘さ | ablation 不足、ベースライン不適切 | 🔬実験屋が再現可能な計画を設計 |

### コード実装フェーズの課題

| 課題 | 現状 | AI Orchestra による解決 |
|---|---|---|
| 数式とコードの乖離 | 論文の式と実装が微妙にずれている | 🧮アルゴリズム検証が数式↔コード対応を確認 |
| 再現性の欠如 | seed未固定、環境依存で結果が変わる | 🔬再現性リーダーがチェックリストで検証 |
| スパゲッティ化 | 研究コードは「動けばいい」で放置 | 📐設計リーダーが構造改善を提案 |
| ボトルネック放置 | GPU使用率40%で学習に何日もかかる | 🤖性能リーダーが最適化箇所を特定 |
| 引き継ぎ不能 | 他者が読めないコード | 📝可読性リーダーが命名・docstring を整備 |

## 1.3 対象ユーザー（研究者）

### プライマリユーザー

- 機械学習・信号処理・コンピュータビジョン等の研究者
- 日常的にアルゴリズム設計と実験を行っている
- Python（PyTorch, NumPy 等）で研究コードを書いている
- 社内 LAN からKotoBuddy API にアクセスできる環境にいる

### 想定する利用シーン

```
朝: 新しいアイデアを思いついた
→ python main.py idea "点群のGNNで〇〇できないか"
→ 5分後に技術的洞察・穴・実験計画が手元に

昼: 実験結果が出た。予想と違う。
→ python main.py idea --follow-up <session_id> --attach results.csv "なぜこうなった？"
→ 前回の議論を踏まえて原因分析・次の仮説が提案される

夜: コードがだいぶ育った。投稿前に整理したい。
→ python main.py review --focus pre_submission ./src/
→ 数式の正しさ、再現性、コード構造がまとめてチェックされる
```

### expertise レベル

ユーザーの専門レベルに応じて AI の発言深度が変わります。

| レベル | 対象者 | AI の振る舞い |
|---|---|---|
| `beginner` | 他分野から来た人、学部生 | 専門用語に括弧で説明。直感的な表現を優先 |
| `intermediate` | 修士学生、分野の基礎を知る人 | 基本概念の説明は省略。計算量の議論は自然に |
| `expert` | 博士学生、ポスドク、当該分野の研究者 | 説明不要。数式を躊躇なく使用。未発表着想レベルの議論 |

## 1.5 2つの機能の位置づけ

AI Orchestra は同一のコアエンジン（指揮者 + 議論 + 評価）を共有する2つの機能を提供します。

```
                    ┌─────────────────────────┐
                    │      共通コアエンジン      │
                    │  Orchestrator / Conductor │
                    │  Agent / Memory / Eval    │
                    └─────────┬───────┬────────┘
                              │       │
              ┌───────────────┘       └───────────────┐
              ▼                                       ▼
┌──────────────────────────┐          ┌──────────────────────────┐
│  機能①: 技術議論          │          │  機能②: コードレビュー     │
│  (idea_discussion.py)    │          │  (code_review.py)        │
│                          │          │                          │
│  入力: テーマ (テキスト)   │          │  入力: フォルダパス        │
│  出力: 洞察・仮説・実験計画│          │  出力: 課題一覧・修正指示書│
└──────────────────────────┘          └──────────────────────────┘
```

---

### 1.5.1 機能①: 技術議論（アイデアブラッシュアップ）

**目的**: 研究テーマやアルゴリズムのアイデアについて、複数 AI が多角的に議論し、技術的洞察・問題点・実験計画を導出する。

**典型的な入力例**:
```bash
python main.py idea "点群データからの特徴量抽出にGNNを使う際の設計指針"
python main.py idea "VAEの潜在空間で異常検知する時、次元数はどう決める？"
python main.py idea "自己教師あり学習でラベルなし時系列データから特徴を抽出したい"
```

**特徴**:
- 研究室でのホワイトボード前の議論を再現
- 理論 → 穴探し → 実装確認 → 実験計画 の流れが自然に生まれる
- 仮説テーブルで「検証すべきこと」を構造化
- `--follow-up` で実験結果を踏まえた継続議論が可能

---

### 1.5.2 機能②: コードレビュー

**目的**: 研究コードのフォルダを入力として、アルゴリズムの正しさ・再現性・性能・設計・可読性・結果整合性の6観点で多角的にレビューし、修正指示書を生成する。

**典型的な入力例**:
```bash
python main.py review ./src/
python main.py review --focus pre_submission ./experiments/
python main.py review --focus performance ./src/model/
```

**特徴**:
- 6体のパートリーダーが分担して調査
- パートリーダー間で相互質問し、観点を横断的に検証
- 全体会議で修正の優先順位を決定
- 最終出力の `vibe_coding_prompt.md` はそのままコーディング AI に渡せる

## 1.6 設計方針

### 1.6.1 CLI ファースト

初期版は**コマンドラインのみ**で動作します。

```bash
# これだけで5分の技術議論が回る
python main.py idea "テーマ"
```

**理由**:
- 研究者はターミナルに慣れている
- 実験スクリプトのパイプラインに組み込みやすい
- UI の開発コストを後回しにして、コアロジックに集中できる

**CLI の UX 設計**:
- `rich` ライブラリによる進捗バー・テーブル・パネル表示
- リアルタイムで AI の発言がターミナルに流れる
- 実行前に計画表示 → ユーザー確認 → 実行 の3ステップ

---

### 1.6.2 拡張可能性（介入機能、UI 等の将来対応）

初期版では実装しないが、**拡張しやすいインターフェースを確保**しておきます。

```python
# 介入ハンドラ: 初期版は NoIntervention（全自動）
# 将来版は CLIIntervention → WebIntervention と差し替え可能
class InterventionHandler(ABC):
    @abstractmethod
    def check_intervention(self, round_num: int, context: dict) -> Optional[str]:
        """各ラウンド間で人間の介入をチェック"""
        pass
```

**将来の拡張ロードマップ**:

| 時期 | 拡張内容 | 対応するインターフェース |
|---|---|---|
| v1.0 | CLI のみ、全自動 | `NoIntervention` |
| v1.1 | CLI で途中介入 | `CLIIntervention` |
| v2.0 | Web UI (Streamlit) | `WebIntervention` + REST API |
| v2.1 | セッション検索・知見 DB | `SessionRepository` |
| v3.0 | 外部ツール連携（Web検索等） | `ToolCallHandler` |

**設計上の工夫**:
- 全てのフェーズ間にフックポイントを設置
- DiscussionEngine はイベントドリブンで設計し、Observer パターンで UI / ログ / 介入を分離
- 設定は全て `settings.yaml` に集約し、コード変更なしで挙動を調整可能

### 1.6.3 研究者向けの会話トーン

AI Orchestra の会話は**研究室での雑談混じりの真剣な議論**を再現します。

**ビジネスミーティングではない**:
```
❌ 「本提案の市場規模は約〇〇億円と試算され、ROIの観点から…」
❌ 「ステークホルダーへの説明責任として…」
```

**研究室のホワイトボード前**:
```
✅ 「それO(N²)じゃない？10万点だと現実的じゃないよ」
✅ 「ちょっと待って、その仮定が崩れるケース考えた？」
✅ 「あー、それ (Wang+2019) のDGCNNがやってた話と同じ構造では」
✅ 「面白い。定式化するとこうなるよね…」
```

**発言ルール**:
- 1発言 50〜150文字。短く鋭く。
- 数式はテキスト表現で自然に混ぜる（`O(N log N)`, `∑`, `∈`）
- 論文引用は `(著者+年)` で簡潔に
- 「たしかに」「でもさ」「ちょっと待って」「あ、それいいね」等の自然な接続詞
- ビジネス観点（ROI、市場性等）には言及しない
- 計算量・収束性・再現性を常に意識した発言

この設計により、出力される `full_conversation.md` は**読み物として面白い**ものになります。研究者が後から読み返して「あ、この視点抜けてた」と気づける体験を目指します。
